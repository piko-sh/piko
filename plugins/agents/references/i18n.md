# Internationalisation (i18n)

Use this guide when adding multi-language support, translations, or locale routing.

## i18n block

Define translations in the `<i18n>` section of a `.pk` file. Only `lang="json"` is honoured; other lang values are silently ignored:

```piko
<i18n lang="json">
{
  "en": {
    "greeting": "Hello",
    "nav": {
      "home": "Home",
      "about": "About"
    }
  },
  "fr": {
    "greeting": "Bonjour",
    "nav": {
      "home": "Accueil",
      "about": "À propos"
    }
  }
}
</i18n>
```

Multiple `<i18n>` blocks per file are allowed (they merge). Nested keys flatten to dot notation: `nav.home`.

## Accessing translations

`r.T(key)` and `r.LT(key)` return a `*piko.Translation` (alias for `*i18n_domain.Translation`). The builder is fluent: typed setters chain and `String()` terminates. In templates, `fmt.Stringer` resolves the value automatically.

### In templates

```piko
<!-- Global translation; falls back to local <i18n> block if not in global files -->
<h1>{{ T("greeting") }}</h1>
<nav>{{ T("nav.home") }}</nav>

<!-- Local-only lookup -->
<p>{{ LT("page-specific-string") }}</p>

<!-- With variables -->
<p>{{ T("welcome").StringVar("name", state.User) }}</p>

<!-- With pluralisation -->
<p>{{ T("items").Count(state.Count) }}</p>
```

### In Go code

```go
func Render(r *piko.RequestData, props piko.NoProps) (Response, piko.Metadata, error) {
    greeting := r.T("greeting", "Hello").String()    // T with fallback
    welcome := r.T("user.welcome").
        StringVar("name", user.Name).
        IntVar("count", inboxCount).
        String()

    return Response{Greeting: greeting, Welcome: welcome}, piko.Metadata{
        Title: r.LT("page.title").String(),
    }, nil
}
```

`Metadata.Title`, `Metadata.Description`, and `Metadata.CanonicalURL` are plain strings, so call `.String()` to terminate the builder before assigning.

## Translation functions

| Function | Lookup order | Use when |
|---|---|---|
| `T(key, fallback...)` | global store first, then local `<i18n>` block | the key is global or may exist in either store |
| `LT(key, fallback...)` | local `<i18n>` block only | the key is page-scoped and you want to ensure no global key shadows it |

Both accept a single optional fallback string. If the key is missing in every store and no fallback is provided, the literal key is returned.

## Placeholder syntax

Use `${expression}` inside translation strings (NOT `{{name}}` and NOT `{name}`):

```json
{
  "en": {
    "welcome": "Welcome, ${name}!",
    "summary": "${user} has ${count} items"
  }
}
```

The expression supports the full Piko expression syntax (operators, property access).

Bind values with typed setters on `*Translation`:

```go
r.T("welcome").StringVar("name", user.Name)
r.T("summary").StringVar("user", user.Name).IntVar("count", n)
```

## Linked messages

Reference another translation key inside a value with `@key.path`:

```json
{
  "en": {
    "common": { "appName": "MyApp" },
    "welcome": "Welcome to @common.appName, ${name}!"
  }
}
```

Recursion is capped at depth 10. Bound variables flow through to the linked translation.

## Backslash escaping

Use `\$` and `\@` to escape literal `$` and `@`:

```json
{
  "en": {
    "email": "support\\@example.com",
    "price": "\\${amount}"
  }
}
```

## Variable setters on `*Translation`

| Method | Type | Locale-aware? |
|---|---|---|
| `StringVar(name, value)` | `string` | no |
| `IntVar(name, value)` | `int` | no |
| `FloatVar(name, value)` | `float64` | yes (decimal separator, grouping) |
| `DecimalVar(name, value)` | `maths.Decimal` | yes |
| `MoneyVar(name, value)` | `maths.Money` | yes (currency symbol position, separators) |
| `BigIntVar(name, value)` | `maths.BigInt` | yes |
| `TimeVar(name, value)` | `time.Time` | yes (medium style for the locale) |
| `Var(name, value)` | `any` | no (uses `fmt.Stringer` or `%v`) |
| `Count(n)` | `int` | yes (CLDR plural form selection; auto-binds `${count}`) |

## Pluralisation

Use pipe-separated forms in a single translation value:

```json
{
  "en": {
    "items": "no items|one item|${count} items"
  }
}
```

```go
r.T("items").Count(n).String()
```

CLDR rule sets cover English (one, other), French (one for 0+1, other), Slavic (one, few, many), Polish (one, few, many with distinct rules), Arabic (zero, one, two, few, many, other), and Asian languages (single form). Escape literal pipes as `||`.

## Locale routing strategies

Configure in `config.json`:

```json
{
  "i18n": {
    "defaultLocale": "en",
    "strategy": "prefix_except_default",
    "locales": ["en", "fr", "de"]
  }
}
```

| Strategy | URLs |
|---|---|
| `prefix` | `/en/about`, `/fr/about` |
| `prefix_except_default` | `/about`, `/fr/about` |
| `domain` | `example.com` (en), `fr.example.com` (fr) |

`Strategy` is a plain string; there is no `piko.StrategyXxx` constant.

## Current locale

```go
locale := r.Locale()              // "fr-CA"
defaultLocale := r.DefaultLocale() // "en"
```

## Per-page opt-in

Multi-locale routing is page-scoped. A page enables it by declaring `SupportedLocales`:

```go
func SupportedLocales() []string {
    return []string{"en", "fr", "de"}
}
```

Without this function, the page produces a single route under the default locale even when other pages opt in.

## Locale-aware links

Use `<piko:a>` for locale-prefixed `href` rewriting:

```piko
<piko:a href="/about">About</piko:a>
<!-- Renders /about under en, /fr/about under fr -->
```

For a language switcher that points at the same path under a specific locale, use `data-locale`:

```piko
<piko:a href="" data-locale="en">English</piko:a>
<piko:a href="" data-locale="fr">Français</piko:a>
```

## Locale-aware formatting outside translations

`r.LF(value)` returns a `*FormatBuilder` for standalone numeric and temporal formatting:

```piko
{{ LF(state.Price).Precision(2) }}        <!-- 19.99 -->
{{ LF(state.Date).Long().DateOnly() }}    <!-- "2 January 2026" -->
```

Methods chain: `Short`, `Medium`, `Long`, `Full`, `DateOnly`, `TimeOnly`, `UTC`, `Precision(n)`, `Locale(code)`. Accepts `time.Time`, numeric types, `maths.Decimal`/`Money`/`BigInt`, `time.Duration`, and `string`.

## Fallback chain

When a key is missing, the lookup walks:

1. The exact requested locale (e.g. `en-GB`).
2. The base language (e.g. `en`).
3. `defaultLocale` from `i18n` config.
4. The fallback string passed as the second arg to `T`/`LT`.
5. The literal key.

`r.T()` consults global first, then local. `r.LT()` consults only local.

## SEO metadata

`piko.GenerateLocaleHead` produces canonical and `hreflang` alternate-link metadata. Returns three values (locale, canonicalURL, alternateLinks):

```go
_, canonical, alternates := piko.GenerateLocaleHead(
    r,
    piko.I18nConfig{
        DefaultLocale: "en",
        Strategy:      "prefix_except_default",
        Locales:       []string{"en", "fr", "de"},
    },
    "/about",
    SupportedLocales(),
)

return Response{}, piko.Metadata{
    Title:          props.PageTitle,
    CanonicalURL:   canonical,
    AlternateLinks: alternates,
}, nil
```

## LLM mistake checklist

- Using `T()` when you mean `LT()` (and vice versa) — `T` walks global→local, `LT` is local-only.
- Forgetting `lang="json"` on `<i18n>` (other values are silently ignored).
- Using `{{name}}` or `{name}` instead of `${name}` for placeholders.
- Treating `r.T(...)` as if it returns `string` — it returns `*Translation`. Call `.String()` when you need a `string` (e.g. assigning to `Metadata.Title`). Templates resolve it implicitly.
- Passing variables as a `map[string]any` — that form does not exist. Use the typed `*Var` setters.
- Mixing up `maths.NewMoneyFromString` arg order (it is `(amount, code)`, not `(code, amount)`).
- Using a non-existent `piko.StrategyPrefix*` constant — the field is a plain string.
- Forgetting to declare `SupportedLocales()` on a page that needs multi-locale routing.
- Hardcoding locale prefixes in URLs instead of using `<piko:a>`.
